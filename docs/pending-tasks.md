## P0 — 核心功能缺失（必须做）

### P0-1: 跨设备清单对比

| 项目 | 说明 |
|------|------|
| **需求来源** | 规范 §6.1 — Downloads 页 |
| **描述** | 下载中心默认只显示"旧电脑有 + 新电脑没有"的软件，而不是全部列表。用户换新电脑后登录，系统自动对比本地扫描结果与云端旧电脑清单，列出需要下载的软件。 |
| **实现思路** | 后端新增 `GET /inventories/compare?other_id=xxx` 接口，返回两设备之间的差异列表（本机有旧机无 / 旧机有本机无）。前端 Downloads 页默认切换到"旧机有、本机无"的过滤模式。 |
| **涉及文件** | `backend/src/routes/inventories.js` (新增 compare 路由), `desktop/src/api/client.ts` (新增 compare 方法), `desktop/src/pages/Downloads.tsx` (默认筛选逻辑) |
| **预估工时** | 2-3 小时 |

## P1 — 重要功能（建议做）

### P1-1: 同步状态指示器

| 项目 | 说明 |
|------|------|
| **需求来源** | 规范 §6.1 — Dashboard |
| **描述** | 在仪表盘显示当前清单的同步状态：`本地仅存` / `已上传云端` / `未登录`。让用户知道数据是否已经安全同步到服务器。 |
| **实现思路** | 扫描时记录本地存储状态，上传成功后更新状态标识。Dashboard 组件根据 `localStorage` 中的 `lastSyncId` 判断并展示。 |
| **涉及文件** | `desktop/src/pages/Dashboard.tsx` |
| **预估工时** | 1 小时 |

### P1-2: 刷新 Token 接口

| 项目 | 说明 |
|------|------|
| **需求来源** | 规范 §4.1 |
| **描述** | JWT 7 天过期后，通过刷新接口换新 Token，避免频繁重新登录。 |
| **实现思路** | 后端新增 `POST /auth/refresh` 路由，接收当前 Token，验证有效后签发新 Token。前端在请求 401 时自动尝试刷新。 |
| **涉及文件** | `backend/src/routes/auth.js`, `desktop/src/api/client.ts` |
| **预估工时** | 1 小时 |

### P1-3: 管理员审核页面

| 项目 | 说明 |
|------|------|
| **需求来源** | 规范 §4.1 — 下载链接审核 |
| **描述** | 后端已有 `/downloads/links/pending` 和 `/downloads/links/:id/verify` 接口，但缺少前端管理页面。管理员登录后应有审核面板，查看待审核链接并一键通过/拒绝。 |
| **实现思路** | 新增 `src/pages/Admin.tsx` 和管理员路由。Dashboard 增加"管理"入口（仅 admin 可见）。 |
| **涉及文件** | `desktop/src/pages/Admin.tsx`, `desktop/src/App.tsx` |
| **预估工时** | 2 小时 |

| **需求来源** | 规范 §6.1 — Downloads |
| **描述** | 在下载中心添加"复制所有下载链接"按钮，一键把所有已匹配软件的官方下载地址复制到剪贴板，方便在浏览器中批量下载。 |
